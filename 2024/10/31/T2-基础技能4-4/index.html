什么是数据包和帧

数据包和帧都是数据的较小单元，它们组合在形成完整的信息或消息。在 OSI 模型中，数据包和帧有不同的定义：帧位于第2层（数据链路层），不包含IP地址等网络层信息

这个过程称为封装（encapsulation）。在封装阶段，涉及 IP 地址的信息称为“数据包”。当信息从数据包剥离时，为“帧”。

数据包是在联网设备之间传输数据的有效方式，因为数据以小块形式交换，减少了网络阻塞的风险。数据包的结构因协议而不同，网络中有许多标准和协议定义了数据包的处理方式，以确保互联网中的数十亿设备能够互通。

每个数据包包含一个头部信息，用于附加传输数据之外的必要信息，以便在网络中有效传输。

| 头部 | 描述 |
| —————— | ————— ——————————————- |
| 生存时间（Time to Live） | 设置了数据包的过期时间，以防止其在无法到达主机或脱离网络时堵塞网络 |
| 校验和（Checksum） | 此字段为 TCP/IP 等协议检查数据完整性。如果数据发生了更改，该值与预期值不同，数据会被视为损坏 |
| 源地址（Source Address） | 数据包发送设备的 IP 地址，用于确定数据返回的路径 |
| 目标地址（Destination Address） | 数据包目标设备的 IP 地址，以便数据知道该去往的目标位置 |

TCP/IP 协议模型（三次握手）

传输控制协议（TCP）是网络中使用的另一种协议模型。TCP/IP 协议模型由四层组成。这四层分别是：

应用层
传输层
网络层
网络接口层

当数据通过 TCP/IP 模型的各层时，信息会逐层添加，称为封装（encapsulation），而逆过程则称为解封装（decapsulation）。

TCP 的定义特征是基于连接的，在数据传输之前，TCP 必须在客户端和服务器之间建立连接。TCP 通过 “三次握手”（Three-way handshake）过程来确保连接的建立。

TCP 的优点 TCP的缺点
保证数据的完整性 需要在设备之间建立可靠连接。如果某个数据块未接收，则需要重新发送整个数据块。
同步数据顺序，防止数据错乱 慢速连接可能导致另一设备的瓶颈，因为连接会一直保持
提高了传输可靠性 由于需要更多计算工作，TCP 比 UDP更慢

TCP 数据包包含多个通过封装添加的头部字段：

头部 描述
源端口（Source Port） 发送方用于发送TCP数据包的端口号，在0-65535随机选择一个未被占用的端口
目标端口（Destination Port） 远程主机上应用或服务的端口号，指定数据包的目标端口
源 IP（Source IP） 发送数据包设备的IP地址，以便数据知道返回到哪里
目标 IP（Destination IP） 数据包目标设备的IP地址，以便数据知道要传输到哪里
序列号（Sequence Number） 连接建立时第一个数据块会被赋予随机序号，用于标识数据的顺序
确认号（Acknowledgement Number） 用于确认接收的数据，下一块数据的序列号为 “当前序列号 + 1”
校验和（Checksum） 确保数据完整性的校验值。接收端通过计算校验和验证数据是否损坏
数据（Data） 包含传输的实际数据内容
标志（Flag） 控制 TCP 握手和数据传输过程的标志位，特定标志触发特定行为

“三次握手” 是 TCP 在两台设备间建立连接的过程，使用特定的消息类型进行通信：

步骤 消息类型 描述
1 SYN 客户端发送的初始 SYN 数据包，用于启动连接并同步两台设备
2 SYN/ACK 服务器响应 SYN 数据包，并发送 SYN/ACK 数据包，确认来自客户端的同步尝试
3 ACK 客户端或服务器发送 ACK 数据包，确认数据包已被成功接收，连接正式建立
4 DATA 一旦连接建立，数据通过 DATA 发送
5 FIN 通信完成后发送 FIN 数据包，关闭连接
— RST 突然结束所有通信，表示连接出现问题

在 TCP 三次握手过程中，任何发送的数据都会被赋予一个初始序列号（ISN），通过 “序列号 + 1” 的方式来确保数据按正确顺序组装。双方必须在序列号上达成一致，以确保数据按正确的顺序发送数据：

SYN - 客户端：客户端发送 SYN 数据包，包含它的初始序列号（ISN = 0），用于同步
SYN/ACK - 服务器：服务器发送 SYN/ACK 数据包，包含它的初始序列号（ISN = 5000），同时收到客户端的初始序列号（ISN = 0）
ACK - 客户端：客户端发送ACK数据包，确认服务器的初始序列号（5000），并包含客户端的序列号加1（0 + 1 = 1）
数据传输开始 - 当服务器发送下一个数据包时，服务器的序列号将是5001

双方通过交换和确认各自的初始序列号来同步数据的顺序，从而确保数据可靠传输

设备 初始序列号（ISN） 最终序列号
客户端（发送方） 0 0 + 1 = 1
服务端（接收方） 5000 5000 + 1 = 5001
客户端（发送方） 1 1 + 1 = 2
服务端（接收方） 5001 5001 + 1 = 5002
客户端（发送方） 2 2 + 1 = 3
服务端（接收方） 5002 5002 + 1 = 5003

TCP 关闭连接

TCP 连接的关闭过程确保设备在数据传输完成后释放系统资源。当设备确认另一设备已成功接收所有数据后，TCP 会启动关闭连接节省系统资源

要关闭 TCP 连接，设备会向另一设备发送一个 FIN 包（Finish包），表示希望终止连接。接收设备会发送一个确认 ACK 包确认这个FIN 请求。关闭连接的过程需双方都发送并确认 FIN 包，以确保连接安全结束

UDP/IP

用户数据报协议（UDP）是轻量级的协议。与 TCP 不同，UDP 是无状态协议，不需要在两台设备之间建立和维持持续连接来发送数据，因此没有三次握手和同步机制

UDP 用于能容忍少量数据丢失的应用场景，或在连接不稳定情况下仍可正常工作的场景。优先考虑传输速度而非数据的完全可靠性。

UDP的优点 UDP的缺点
速度比 TCP 快，因为不用建立连接 UDP 不关心数据是否成功接收，不保证数据可靠性
让应用程序决定是否控制数据包的发送速度 对于软件开发人员来说灵活
不保留持续连接，减少了系统资源消耗 连接不稳定时用户体验变差，UDP 不提供重传或纠错机制

UDP 在两台设备之间传输数据时无需建立连接，不关心数据是否成功接收，也没有数据完整性保障。结构更简单。UDP是无状态的，连接期间不会发送任何确认与TCP。共享一些相同的头部字段，但头部字段较少：

头部字段 描述
生存时间（TTL） 设置数据包的过期时间，防止未达目标或离开网络时堵塞网络
源地址（Source Address） 数据包发送设备的IP地址，以便数据知道返回到哪里
目标地址（Destination Address） 数据包的目标设备IP地址，以便数据知道要传输到哪里
源端口（Source Port） 发送方发送 UDP 数据包的端口号，在0-65535之间未占用端口中随机选择
目标端口（Destination Port） 接收设备上应用程序或服务的端口号
数据（Data） 用于存储传输的数据内容

端口 101

端口是数据交换的重要接口，端口定义了网络设备如何传输和接收特定类型的数据。建立连接后，设备发送和接收的任何数据都会通过端口。端口号是 0 到 65535 之间的数值，不同的端口号被分配给特定的服务和协议

由于端口范围是 0 到 65535 ，如果没有规则会导致混乱，不清楚哪个应用程序在使用哪个端口。一套标准规则将特定端口分配给特定的应用程序和协议。所有的网页浏览器都遵循一个共同的标准，数据通过端口80发送

HTTP 协议使用的标准端口是 80，但其他协议也有分配了标准端口。介于 0 到 1024 之间的端口为常用端口，这些端口将会分配给常用协议

协议 端口号 描述
文件传输协议（FTP） 21 用于文件共享的客户端，服务器模型协议，允许下载和上传文件
安全外壳协议（SSH） 22 通过基于文本的界面安全登录系统，用于远程系统管理
超文本传输协议（HTTP） 80 支持WWW万维网的协议，浏览器使用此协议下载文本，图片和视频
超文本传输协议安全版（HTTPS） 443 与 HTTP 相同，但通过加密实现更高的安全性
服务器消息块协议（SMB） 445 类似于 FTP，还允许共享设备以及文件
远程桌面协议（RDP） 3389 可视化的桌面界面实现远程登录，便于远程控制计算机
