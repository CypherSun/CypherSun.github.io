CIA

在评估某个事物的安全性之前，需要深入理解其组成要素。通常，在评估系统安全性时，从机密性、完整性和可用性这三个方面出发，即 CIA 三角模型

CIA 三角模型：
机密性（Confidentiality）：确保只有授权人员或指定接收者能够访问数据，防止未经授权的泄露。
完整性（Integrity）：确保数据不会在传输或存储过程中被篡改，同时能检测出任何篡改的痕迹。
可用性（Availability）：确保系统或服务在需要时可供使用，防止因系统宕机或其他故障导致的不可用情况。

不同的系统侧重点不同

CIA 之外的安全要素

在 CIA 模型之外，包括真实性和不可抵赖性：
真实性：确保数据非伪造，数据确实来自声称的来源
不可抵赖性：确保信息来源无法否认自己是数据的提供者

Parker 六元组：

1998 年，Donn Parker 提出了 Parker 六元组，将安全性扩展为六个要素：

可用性
实用性：关注信息的有效可用性。
完整性
真实性
机密性
拥有性：即信息的控制权。


DAD

系统的安全性可能通过多种方式受到攻击，主要表现为：机密数据的泄露、数据的篡改或数据的破坏。这些威胁对应于 CIA三要素的对立面，被称为 DAD 三要素。

泄露（Disclosure）：即机密性的对立面，指未经授权的数据访问或泄露。
篡改（Alteration）：即完整性的对立面，指数据被非法修改或操控。
破坏/拒绝服务（Destruction/Denial）：即可用性的对立面，指数据或系统因攻击或故障而不可用或被破坏。

DAD 三要素即 CIA 三要素的对立面，防止数据的泄露、篡改和破坏（拒绝服务）是确保机密性、完整性和可用性的核心目标。良好的安全实践需要在三者之间找到平衡，因为过度强调机密性和完整性可能会妨碍系统的可用性，而一味增加可用性可能会降低机密性和完整性。平衡 CIA 三要素有助于构建安全、稳定的系统。

安全模型的基本概念

安全三要素包括机密性、完整性和可用性（CIA）。为了创建能够确保一种或多种安全功能的系统，使用不同的安全模型。

Bell-LaPadula模型

Bell-LaPadula模型主要通过以下三条规则来确保系统的机密性：

简单安全属性（Simple Security Property）：“不可向上读取”（no read up），即低安全级别的主体不能读取高安全级别的对象，从而防止未授权主体访问敏感信息。

星安全属性（Star Security Property）：“不可向下写入”（no write down），即高安全级别的主体不能向低安全级别的对象写入数据，避免敏感信息泄露。

自由裁量安全属性（Discretionary Security Property）：通过访问矩阵控制读写操作：

主体 对象A 对象B
主体1 写 无访问
主体2 读/写 读

前两条规则也可以总结为“向上写入，向下读取”（write up, read down）：可以将机密信息与更高安全级别的人共享（向上写入），并从较低安全级别的人接收信息（向下读取）。

Bell-LaPadula 模型在处理文件共享存在局限性

Biba 模型

Biba 模型旨在通过以下两条规则确保系统的完整性：

简单完整性属性（Simple Integrity Property）：“不可向下读取”（no read down），即高完整性主体不应从低完整性对象中读取信息，以避免受到低完整性数据的污染。

星完整性属性（Star Integrity Property）：“不可向上写入”（no write up），即低完整性主体不应向高完整性对象写入信息，以防止污染高完整性数据。

这两条规则总结为“向上读取，向下写入”（read up, write down），与 Bell-LaPadula 模型的“向上写入，向下读取”相反。Biba 模型侧重于完整性，而 Bell-LaPadula 模型则侧重于机密性。

Biba 模型在防范内部人员威胁方面存在局限性

Clark-Wilson 模型

Clark-Wilson 模型同样通过以下概念实现完整性：

受限数据项（CDI）：需要保持完整性的数据类型
非受限数据项（UDI）：不要求完整性约束的其他数据类型
转换过程（TPs）：用于维护 CDI 完整性的操作过程
完整性验证过程（IVPs）：用于检查和验证 CDI 的有效性，确保其未被篡改


纵深防御

纵深防御是一种通过多层次保护措施来提升系统安全性的策略，也称为多层次安全（Multi-Level Security）。这种方法通过多重防御层抵御不同的威胁，使攻击者无法通过单一层次的安全缺口进入系统。每一层都设计有不同的控制措施，以最大限度地提高整体安全性。纵深防御的核心理念是，即使一层安全防护被突破，后续的层级仍能为系统提供进一步保护。

ISO/IEC 19249 标准

ISO/IEC 19249标准由国际标准化组织（ISO）和国际电工委员会（IEC）制定，名称为“信息技术 - 安全技术 - 安全产品、系统和应用的架构和设计原则目录”。其目的是为安全架构提供指导性建议，以帮助实现更高的系统安全性。

ISO/IEC 19249列出以下五项架构原则：

域分离（Domain Separation）：将相关组件分组为独立实体，每个域拥有一组统一的安全属性

分层（Layering）：系统划分为多个抽象层，使安全策略可在不同层级实施并便于验证。分层通常与纵深防御原则有关。

封装（Encapsulation）：通过特定方法隐藏底层实现，防止不当访问。在面向对象编程中，封装确保对象通过预定义的接口操作数据。

冗余（Redundancy）：保障可用性和完整性。

虚拟化（Virtualization）：将硬件资源分配给多个操作系统，使其隔离运行，提供沙盒功能。虚拟化不仅支持云环境，还可用于安全观察恶意程序，增强安全边界。

ISO/IEC 19249的五项设计原则

最小权限：“基于需要”或“知情需要”原则，最小权限原则要求只授予完成任务所需的最低权限，避免多余权限。

攻击面最小化：系统存在已知和未知漏洞，这些漏洞构成潜在风险。为减少风险，应尽可能缩小攻击面。

集中化参数验证：许多威胁来自于无效输入的利用，可能导致拒绝服务或远程代码执行等风险。参数验证是系统安全的关键步骤，为简化管理和提高效率，应将参数验证集中在一个库或系统中进行。

集中化通用安全服务：集中化管理安全服务可提高安全性与效率。

准备错误和异常处理：系统应为错误和异常事件做好准备。安全设计应包括安全失败模式，同时确保错误信息不泄露敏感信息


零信任与信任但验证

在信任方面，两条安全原则值得关注：

信任但验证
该原则强调，即便信任某个实体，也应始终进行验证。验证通常通过日志机制完成，定期检查日志确保操作正常。由于手动验证难以实现全面监控，因此需要自动化的安全机制，以便及时检测异常行为。

零信任
零信任原则将信任视为一种潜在脆弱性，特别注重防范内部威胁。零信任的核心理念是“永不信任，始终验证”，将所有实体视为潜在威胁，只有在经过验证后才授予信任。与传统模式不同，零信任不会因设备位置或归属而授予信任，而是在每次访问资源前均需进行身份验证和授权。通过这种方式，即使出现安全漏洞，零信任架构也能有效将损害控制在最小范围。

零信任架构的一种实现方式是微分段（Microsegmentation）。微分段将网络划分为极小的单元，甚至小到单一主机。各分段之间的通信均需身份验证、访问控制检查等多重安全措施，以确保高安全性。

尽管零信任的实施需确保不对业务产生负面影响，但在可行的情况下，零信任原则应尽可能应用。

威胁与风险

漏洞（Vulnerability）：指系统或资产的弱点或易受攻击的特性，可能被利用而导致损害。
威胁（Threat）：指可能利用漏洞的潜在危险，包含攻击者行为或自然事件等。
风险（Risk）：指威胁利用漏洞的可能性及其对业务产生的潜在影响。
