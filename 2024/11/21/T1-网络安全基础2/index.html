CIA

在讨论某物安全之前，需要明确安全的核心构成。当评估系统的安全性时，应重点关注安全三要素：机密性、完整性和可用性（CIA）。

机密性（Confidentiality）：确保数据仅被授权访问
完整性（Integrity）：确保数据未被篡改，且任何篡改都可被发现
可用性（Availability）：确保系统或服务在需要时可用

在线购物例子：
机密性：顾客希望信用卡号仅被支付处理机构妥善保管。如果支付信息可能会被泄露，顾客可能会到其他平台购物。并且一旦泄露，公司也会面临重大损失。
完整性：若入侵者篡改订单的送货地址，包裹可能被寄错，顾客将失去对平台的信任。
可用性：若平台网站或应用无法访问，顾客无法下单，可能会到其他平台购物。

医疗系统例子：
机密性：患者病历的非法披露可能导致医疗机构承担法律责任。
完整性：病历被篡改可能引发误诊，危及生命，医院因此失去可信度。
可用性：若医疗系统宕机，医生无法访问病历，可能导致误诊或漏诊。

三要素的重要性因场景而异。例如，大学发布的公告中，机密性可能不重要，但文件的完整性至关重要。

除了 CIA

在 CIA 安全三要素之外，还可以引入以下两个安全要素：

真实性：确保数据或信息来源真实，且非伪造或冒充。
不可否认性：确保数据提供方无法否认自己是该数据的来源。

这两者在实际场景中密不可分。例如，外卖送货时，如果订单是真实的，收件人应能确认自己下单。同时，收件人不能否认下单行为。
在更大的例子中，例如公司接到购买 1000 辆汽车的订单，公司需要确认订单的真实性，同时确保下单方无法否认自己的行为。缺乏这两项保障会导致严重后果。

Parker 六元组模型

1998 年，Donn Parker 提出了一个包含六个安全要素的模型，被称为 Parker 六元组模型。

这六个要素包括：
可用性
效用性
完整性
真实性
机密性
占有性

效用性：关注信息是否对用户有实际用处。例如，如果用户丢失了解密密钥，即使数据完整且存在，用户仍无法使用，信息因此失去了效用性
占有性：关注信息的所有权和控制权。例如，备份硬盘被窃取或数据被勒索软件加密，都会导致用户丧失对信息的占有权

DAD

系统的安全性可能受到多种方式的攻击，包括：机密数据的泄露、数据的篡改和数据的破坏/拒绝服务。

泄露（Disclosure）：泄露是机密性的对立面，即机密数据的未授权披露。
篡改（Alteration）：篡改是完整性的对立面，例如支票被篡改会导致严重后果。
破坏/拒绝服务（Destruction/Denial）：破坏或拒绝服务攻击是可用性的对立面，例如系统被攻击导致无法使用。

这三类攻击对应于 DAD 三要素：泄露、篡改和破坏/拒绝服务。它们与 CIA 三要素：机密性、完整性、可用性一一对应。

患者记录系统例子：
泄露：如果攻击者窃取并公开患者病历，会导致医疗机构在法律、财务和信誉上遭受严重损失
篡改：病历被篡改可能导致错误治疗方案，甚至危及患者生命
破坏/拒绝服务：在医疗机构中，数据库系统的不可用可能使病历无法及时访问，直接影响诊疗工作，甚至导致全面停滞

防范泄露、篡改和破坏攻击是保障系统安全的核心任务，这也意味着需要同时维护机密性、完整性和可用性。
这三者之间往往存在权衡，过度强调机密性和完整性可能降低可用性，而提升可用性可能增加泄露或篡改的风险。因此，制定有效的安全策略需要在三者之间找到最佳平衡点，以满足实际需求。

安全模型的基本概念

安全三要素（CIA）包括机密性、完整性和可用性，使用安全模型能创建一个确保一项或多项安全功能的系统。

Bell-LaPadula 模型

Bell-LaPadula 模型专注于保障机密性，通过以下三条规则实现：

简单安全规则（no read up）：低安全级别的主体不能读取高安全级别的客体，防止低级别用户访问敏感信息
星号安全规则（no write down）：高安全级别的主体不能向低安全级别的客体写入数据，避免敏感信息泄露
任意安全规则：通过访问矩阵控制读写权限

主体 对象A 对象B
主体1 写入 无权限
主体2 读/写 读取

这些规则总结为“向上写、向下读”（write up, read down），即高权限主体可向更高权限主体传递数据，低权限主体可读取来自更低权限的数据。

Bell-LaPadula 模型不适用于所有场景，例如文件共享环境，因为它未考虑完整性和可用性的需求。

Biba 模型

Biba 模型专注于完整性，通过以下规则实现：

简单完整性规则（no read down）：高完整性主体不能读取低完整性客体的信息，防止低完整性数据污染高完整性数据。
星号完整性规则（no write up）：低完整性主体不能向高完整性客体写入信息，防止无效或恶意数据影响高完整性资源。

这些规则总结为“向上读、向下写”（read up, write down），与 Bell-LaPadula 模型形成对比。Biba 模型专注于完整性，而 Bell-LaPadula 模型关注机密性。

Biba 模型无法处理内部威胁，例如来自内部人员的攻击。

Clark-Wilson 模型

Clark-Wilson 模型通过以下概念保障完整性：

受约束数据项（CDI）：需保护完整性的重要数据类型
非受约束数据项（UDI）：无需完整性保护的普通数据类型，例如用户或系统输入
转换过程（TPs）：对 CDI 进行操作的安全过程，需确保操作不会破坏数据完整性
完整性验证过程（IVPs）：用于检查 CDI 是否满足完整性要求

除了上述三种模型，其他安全模型包括：
Brewer and Nash 模型：防止利益冲突，常用于会计或咨询领域。
Goguen-Meseguer 模型：基于非干涉理论，确保不同用户的操作互不影响。
Sutherland 模型：基于信息流控制，防止隐性信息泄露。
Graham-Denning 模型：描述安全权限的管理与分配。
Harrison-Ruzzo-Ullman 模型：处理权限操作的安全性与可达性。

纵深防御

纵深防御是一种通过多个层级构建安全系统的策略，因此也称为多层安全（Multi-Level Security）

假设有一个锁着的抽屉，里面存放着重要文件和贵重物品。在多层安全的理念下，可能会选择更全面的防护措施：抽屉上锁、房间上锁、公寓主门上锁、建筑物大门上锁，并在关键位置安装安全摄像头。

尽管这些措施无法完全阻止所有威胁，但它们可以显著提高防护能力，拦截大部分威胁，并有效延缓攻击者的侵入时间。

ISO/IEC 19249

ISO/IEC 19249:2017《信息技术——安全技术——安全产品、系统和应用的架构与设计原则目录》由国际标准化组织（ISO）和国际电工委员会（IEC）共同制定，其目的是为安全系统的架构和设计提供指导。

ISO/IEC 19249 提出的五项架构原则：

域分离（Domain Separation）：将相关组件划分为独立域，每个域具备统一的安全属性。例如，x86 处理器特权级别中，操作系统内核运行在 ring 0 最高权限，用户模式应用运行在 ring 3最低权限。域分离的理念也被包含在 Goguen-Meseguer 模型中。

分层（Layering）：将系统分为多个抽象层，以便在不同层次实施安全策略，增强验证性。例如，OSI 模型的七层架构中，每层为上一层提供服务，同时便于独立实施和验证安全策略。另一个例子是编程语言通过封装底层磁盘操作为高级函数接口，从而简化访问并减少错误风险。这种分层理念与纵深防御密切相关。

封装（Encapsulation）：封装通过隐藏底层实现，防止对内部数据的直接操作。例如，在面向对象编程（OOP）中，时钟对象可提供 increment() 方法，而非允许用户直接修改秒变量，以避免非法值。在大型系统中，设计合适的应用程序编程接口（API）实现封装，确保访问受控。

冗余（Redundancy）：冗余通过备份机制保障系统的可用性和完整性。例如，硬件服务器中的双电源模块，即使一个失效，系统仍能正常运行。RAID 5 配置可在一块硬盘损坏时通过校验位恢复数据。这种机制同时提高了系统的容错能力。

虚拟化（Virtualization）：虚拟化利用共享硬件资源支持多个操作系统运行在同一硬件上，增强安全边界。它提供沙箱功能，用于隔离恶意程序的执行并安全观测其行为。虚拟化技术已成为云服务中的关键组成部分。

ISO/IEC 19249 提出的五项设计原则

最小权限（Least Privilege）：仅授予完成任务所需的最低权限，即按需授权或基于需要知道的原则。例如，用户需要查看文档时，应只赋予读取权限，而非写入权限。

攻击面最小化（Attack Surface Minimisation）：通过减少系统暴露的潜在风险点，降低攻击可能性。攻击面包括已知和未知漏洞。例如，在使用 Linux 系统时，可以禁用不必要的服务以减少可能被利用的入口点。

集中化参数验证（Centralized Parameter Validation）：用户输入可能引发安全威胁，如拒绝服务或远程代码执行攻击。集中化验证参数有助于确保输入合法，避免重复验证逻辑。例如，将所有输入验证集中在单一库或模块中统一处理。

集中化通用安全服务（Centralized General Security Services）：将安全服务集中化管理以提升整体安全性。例如，使用集中式认证服务器统一管理身份验证，同时采取措施避免单点故障，确保服务可用性。

错误与异常处理准备（Preparing for Error and Exception Handling）：系统应为安全失败（fail-safe）进行设计，以在异常情况下仍能保护安全。例如，当防火墙崩溃时，应默认阻止所有流量，而非允许流量通过。此外，应避免错误信息泄露敏感内容，如内存转储中包含的私人信息。


零信任与信任但验证

信任是必要且复杂的。例如对硬件供应商的不信任则可能会放弃选择其产品。在商业环境中，信任问题更加复杂，因此需要指导性的安全原则。以下是两条核心原则：

信任但验证（Trust but Verify）

即使信任某个实体，如用户或系统，但仍需通过验证确认其行为安全。这通常通过日志记录和检查来实现，例如审查系统运行是否异常。然而，全面验证所有内容并不现实，因此需要依赖自动化工具，如代理服务器、入侵检测系统（IDS）和入侵防御系统（IPS）。这些机制帮助高效验证并减少人工干预。

零信任（Zero Trust）

零信任理念视信任为潜在漏洞，核心是“永不信任，始终验证”。在零信任架构中，任何实体都需经过严格验证，无论其设备位置或归属权，如公司设备。零信任摒弃传统默认信任内部网络的做法，对每次资源访问都要求身份验证和授权，从而有效限制潜在安全漏洞的影响范围。

微分段是零信任的重要实现方式，通过将网络划分为小型分段，甚至单个主机。不同分段间的通信需满足严格的安全要求，如身份验证和访问控制列表（ACL）检查。

尽管零信任可提升安全性，但过度实施可能对业务流程造成负面影响。应根据实际需求平衡安全性与效率，在可行范围内尽可能采用零信任模式。

威胁与风险

在信息安全中，为了避免混淆，需要明确以下三个关键术语：

漏洞（Vulnerability）：系统中易受攻击或损害的弱点
威胁（Threat）：可能利用漏洞引发的潜在危险或风险
风险（Risk）：威胁利用漏洞的可能性及其对业务的潜在影响

例如，汽车展厅的门窗和墙体由普通玻璃制成。玻璃的脆弱性是漏洞，玻璃可能被打碎是威胁。而展厅所有者需要评估风险，玻璃被打碎的可能性有多高，以及这一事件可能对业务造成何种影响。

假设在一家医院工作，其数据库系统存储所有患者的医疗记录。发现新闻中披露该数据库系统存在一个漏洞，并已发布概念验证代码，证明漏洞可以被利用。在这种情况下，需要评估风险的严重性，并制定应对措施，例如立即更新补丁或实施临时缓解策略。
