什么是数据包和帧

数据包和帧都是小块数据，当它们组合在一起时，会形成更大的信息或消息。然而，在 OSI 模型中，它们是两个不同的东西。帧位于第 2 层 - 数据链路层，这意味着没有 IP 地址之类的信息。例如将一个信封放在另一个信封中并将其寄出。第一个信封将是邮寄的数据包，但一旦打开，信封中的信封仍然存在并包含数据，这是一个帧。

这个过程称为封装，假设当谈论任何 IP 地址时，谈论的是数据包。当封装信息被剥离时，谈论的是帧本身。

数据包是在联网设备之间传递数据的有效方式。由于这些数据是以小块的形式进行交换的，因此与一次发送大消息相比，在网络上出现瓶颈的可能性较小。

例如，当从网站加载图像时，该图像不会以整体形式发送到计算机，而是会以小块的形式发送到计算机，然后到达计算机后重新构建以形成最终图像。

数据包具有不同的结构，这取决于发送的数据包类型。网络充满了标准和协议，它们充当了一组规则，规定了数据包在设备上的处理方式。互联网上有数十亿台设备连接在一起，如果没有标准化，事情很快就会崩溃

使用此协议的数据包将具有一组标头，其中包含通过网络发送的数据的附加信息。

一些值得注意的标题包括：

标头 描述
生存时间 如果数据包无法到达主机或逃脱，此字段将为数据包设置一个过期计时器，以免堵塞网络。
校验和 此字段为 TCP/IP 等协议提供完整性检查。如果任何数据发生更改，此值将与预期值不同，因此会损坏。
源地址 发送数据包的设备的 IP 地址，以便数据知道返回到哪里。
目的地址 数据包被发送到的设备的 IP 地址，以便数据知道下一步要传输到哪里。

TCP/IP（三次握手）

传输 控制协议 (TCP) 是网络中使用的另一个规则。

该协议与 OSI 模型非常相似。TCP / IP 协议由四层组成，可以说只是 OSI 模型的总结版本。这些层是：

应用
运输
互联网
网络接口

与 OSI 模型的工作方式非常相似，当数据或数据包穿过 TCP 模型的每一层时，信息都会添加到其中。这个过程称为封装 - 而这个过程的逆过程称为解封装。

TCP 的一个定义特征是它是基于连接的，这意味着 TCP 必须在发送数据之前在客户端和充当服务器的设备之间建立连接 。

因此，TCP 保证发送的任何数据都会在另一端收到。这个过程称为三向握手。下表比较了TCP的优点和缺点：

TCP的优点 TCP的缺点
保证数据的完整性。 需要两个设备之间建立可靠的连接。如果未收到一小块数据，则整个数据块无法使用，必须重新发送。
能够同步两个设备，以防止彼此以错误的顺序被数据淹没。 较慢的连接可能会成为另一台设备的瓶颈，因为该连接将始终保留在另一台设备上。
执行更多流程以确保可靠性 TCP 明显比UDP慢，因为使用该协议的设备必须完成更多的工作或计算。

TCP 数据包包含各种信息部分，这些部分称为标头，是在封装时添加的。在下表中解释一些关键的标头：

标头 描述
源端口 此值是发送方打开的用于发送 TCP 数据包的端口。此值是随机选择的（从 0-65535 中当前未被使用的端口中选择）。
目的港 此值是应用程序或服务在远程主机（接收数据的主机）上运行的端口号；例如，在端口 80 上运行的 Web 服务器。与源端口不同，此值不是随机选择的。
源 IP 这是发送数据包的设备的 IP 地址。
目标 IP 这是数据包目的地设备的 IP 地址。
序列号 当建立连接时，传输的第一部分数据将被赋予一个随机数。
确认号码 当一段数据被赋予序列号后，下一段数据的序列号将会是序列号 + 1。
校验和 此值赋予TCP完整性。进行数学计算并记住输出。当接收设备执行数学计算时，如果输出与发送的不同，则数据必定已损坏。
数据 该文件头是用来存储数据（也就是正在传输的文件的字节）的地方。
旗帜 此标头决定了握手过程中任一设备应如何处理数据包。特定标志将决定特定行为。

三次握手 - 这是用于在两个设备之间建立连接的过程的术语。三次握手使用一些特殊消息进行通信 - 下表重点介绍了主要消息：

步 信息 描述
1 同步 SYN 消息是客户端在握手期间发送的初始数据包。此数据包用于启动连接并同步两个设备（稍后会进一步解释这一点）。
2 同步/确认 该数据包由接收设备（服务器）发送，以确认来自客户端的同步尝试。
3 确认 客户端或服务器可以使用确认包来确认一系列消息/数据包已被成功接收。
4 数据 一旦建立连接，数据（例如文件的字节）将通过“DATA”消息发送。
5 完成 此数据包用于在连接完成后干净地（正确地）关闭连接。
# 恢复时间 此数据包会突然结束所有通信。这是最后的手段，表示在此过程中出现了一些问题。例如，如果服务或应用程序无法正常工作，或者系统出现资源不足等故障。

任何发送的数据都会被赋予一个随机数字序列，并使用该数字序列重建并以 1 为增量。两台计算机必须就相同的数字序列达成一致，才能以正确的顺序发送数据。此顺序在三个步骤中达成一致：

SYN - 客户端：这是SYN初始序列号 (ISN)， 同步为 (0)
SYN/ACK - 服务器：这是SYN初始序列号 (ISN) （5,000），确认初始数字序列（0）
ACK - 客户端：确认初始序列号 (ISN) 为 (5,000)，这里有一些数据，即 ISN+1 (0 + 1)

设备 初始数字序列 (ISN) 最终数字序列
客户端（发送方） 0 0 + 1 = 1
客户端（发送方） 1 1 + 1 = 2
客户端（发送方） 2 2 + 1 = 3

TCP 关闭连接：
快速解释一下 TCP 关闭连接的过程。首先，一旦一个设备确定另一个设备已成功收到所有数据，TCP 就会关闭连接。

由于 TCP 在设备上保留系统资源，因此最佳做法是尽快关闭 TCP 连接。

要关闭 TCP 连接，设备将向另一台设备发送 FIN 数据包。当然，使用 TCP 时，另一台设备也必须确认此数据包。


UDP/IP

用户数据报协议 (UDP) 是另一种用于在设备之间传递数据的协议。

与其 TCP 不同，UDP 是一种无状态协议，它不需要两台设备之间保持持续连接来发送数据。例如，不会发生三次握手，两台设备之间也没有任何同步。

OSI 模型中对这两种协议的一些比较。也就是说，UDP 用于应用程序可以容忍数据丢失的情况，例如视频流或语音聊天或不稳定连接不是最终问题的情况。下表比较了 UDP 的优缺点：

UDP的优点 UDP的缺点
UDP 比 TCP 快得多。 UDP 并不关心数据是否被接收到。
UDP 让应用程序（用户软件）决定是否可以控制数据包的发送速度。 从这个意义上来说，它对于软件开发人员来说非常灵活。
UDP 不像 TCP 那样在设备上保留连续的连接。 这意味着不稳定的连接会给用户带来糟糕的体验

如上所述，在两个设备之间建立连接时不会发生任何过程。这意味着不会考虑是否收到数据，也不会提供任何像 TCP 那样的安全措施，例如数据完整性。

UDP 数据包比 TCP 数据包简单得多，并且标头较少。不过，两种协议都有一些标准标头，如下表所示：

标头 描述
生存时间（TTL） 此字段为数据包设置了一个过期计时器，因此，即使数据包无法到达主机或逃脱，也不会堵塞网络
源地址 发送数据包的设备的 IP 地址，以便数据知道返回到哪里。
目的地址 数据包被发送到的设备的 IP 地址，以便数据知道下一步要传输到哪里。
源端口 此值是发送方打开的用于发送 UDP 数据包的端口。此值是随机选择的，从 0-65535 中当前未被使用的端口中选择。
目的港 此值是应用程序或服务在远程主机或接收数据的主机上运行的端口号；例如，在端口 80 上运行的 Web 服务器。与源端口不同，此值不是随机选择的。
数据 这个头是存储数据，也就是正在传输的文件的字节的地方。

UDP 连接过程与 TCP 连接过程是不同的，UDP 是无状态的。连接期间不会发送任何确认。

港口 101（实用）

港口或许恰如其名，是交换数据的重要地点。在海港和港口，希望停靠在港口的船只必须前往与船只尺寸和设施相符的港口。船只排队后，将停靠在港口的港口 。例如，邮轮不能停靠在为渔船建造的港口，反之亦然。

这些端口强制规定什么可以停放以及停放在哪里 — 如果不兼容，它就不能停放在这里。网络设备还使用端口来在相互通信时执行严格的规则。建立连接后，设备发送或接收的任何数据都将通过这些端口发送。在计算中，端口是介于 0 和 65535 之间的数值。

由于端口范围从 0 到 65535，因此很容易出现无法跟踪哪个应用程序正在使用哪个端口的风险。繁忙的港口总是一片混乱！有一组标准规则将应用程序、软件和行为联系起来。例如，通过强制要求任何 Web 浏览器数据都通过端口 80 发送，软件开发人员可以设计一个 Web 浏览器，如 Google Chrome 或 Firefox 以相同的方式解释数据。

这意味着所有网络浏览器现在都共享一个共同的规则：数据通过端口 80 发送。浏览器的外观、感觉和易用性取决于设计者或用户的决定。

虽然 Web 数据的标准规则是 端口 80 ，但一些其他协议也分配了标准规则。0 到 1024 之间的任何端口都称为公共端口。让在下面探索其中一些其他协议：

协议 端口号 描述
文件传输协议 (FTP) 21 该协议由基于客户端-服务器模型构建的文件共享应用程序使用，这意味着可以从中心位置下载文件。
安全外壳 (SSH) 22 该协议用于通过基于文本的界面安全地登录系统进行管理。
超文本传输协议 (HTTP) 80 此协议为万维网 (WWW) 提供支持，浏览器使用此协议下载网页的文本、图片和视频。
超文本传输安全协议 (HTTPS) 443 该协议与上述协议完全相同；但是，使用加密来安全地进行。
服务器信息块 (SMB) 445 该协议类似于文件传输协议 (FTP)。但是，除了文件之外，SMB 还允许共享打印机等设备。
远程桌面协议 (RDP) 3389 该协议是一种使用可视化桌面界面登录系统的安全方式，与 SSH 协议基于文本的限制相反。

仅简要介绍了网络安全中较常见的协议。

这里值得注意的是，这些协议仅遵循标准。也就是说，可以在标准端口以外的其他端口上管理与这些协议交互的应用程序在 8080 而不是 80 标准端口上运行 Web 服务器。但请注意，应用程序将假定遵循标准，因此必须提供冒号(:) 以及端口号。
