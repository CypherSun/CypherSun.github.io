什么是 HTTP？（超文本传输协议）

HTTP 是浏览网站时使用的协议，由蒂姆·伯纳斯-李及其团队在 1989 到 1991 年间开发。HTTP 定义了与网页服务器进行通信的规则，用于传输网页数据，比如 HTML、图片、视频等内容。

什么是 HTTPS？（安全超文本传输协议）

HTTPS 是 HTTP 的安全版本。HTTPS 数据是加密的，这不仅可以防止他人查看您接收和发送的数据，还能确保您连接的是真正的网页服务器，而不是冒充的服务器。

请求和响应

当我们访问网站时，浏览器需要向网页服务器请求诸如 HTML、图片等资源，并下载相应的响应。在此之前，需要明确告诉浏览器如何以及从哪里获取这些资源，这就是 URL 的作用。

什么是 URL？（统一资源定位符）
如果你使用过互联网，就一定用过 URL。URL 主要是关于如何在互联网上访问资源的指令。下面的图片展示了 URL 的不同部分及其作用（每次请求时不一定会用到所有部分）。

一个示例 URL 的结构图，其中 http 是协议（Scheme），user:password 是用户信息，tryhackme.com 是域名或主机，80 是端口，view-room 是路径，?id=1 是查询字符串，#task3 是片段。完整地址是 http://user:password@tryhackme.com:80/view-room?id=1#task3。

协议（Scheme）： 指示用于访问资源的协议，如 HTTP、HTTPS、FTP（文件传输协议）。
用户（User）： 某些服务需要认证登录，可以在 URL 中加入用户名和密码进行登录。
主机（Host）： 要访问的服务器的域名或 IP 地址。
端口（Port）： 连接的端口，通常 HTTP 为 80，HTTPS 为 443，但可以是 1 - 65535 范围内的任意端口。
路径（Path）： 要访问的资源的文件名或位置。
查询字符串（Query String）： 可以传递给请求路径的额外信息。例如 /blog?id=1 表示请求 ID 为 1 的博客文章。
片段（Fragment）： 对请求页面中某个具体位置的引用。常用于长页面，用户可以通过直接链接到该部分以便立即查看。
发起请求
仅需一行即可向网页服务器发起请求，例如 GET / HTTP/1.1。

然而，为了获得更丰富的网页体验，还需要发送其他数据。这些数据通过头部信息（headers）来传递，头部包含了传递给通信服务器的附加信息，我们将在“头部”任务中详细介绍。

请求示例：
makefile
复制代码
GET / HTTP/1.1
Host: tryhackme.com
User-Agent: Mozilla/5.0 Firefox/87.0
Referer: https://tryhackme.com/
对每行的说明：

第 1 行： 发送 GET 方法（详情请参见 HTTP 方法任务），请求主页 /，并告知服务器我们使用的是 HTTP 1.1 协议版本。
第 2 行： 告诉服务器请求的是 tryhackme.com 网站。
第 3 行： 告诉服务器我们使用的是 Firefox 87 浏览器。
第 4 行： 告诉服务器推荐我们到这个页面的来源是 https://tryhackme.com。
第 5 行： HTTP 请求总是以空行结束，以告知服务器请求已完成。
响应示例：
php
复制代码
HTTP/1.1 200 OK
Server: nginx/1.15.8
Date: Fri, 09 Apr 2021 13:34:03 GMT
Content-Type: text/html
Content-Length: 98

TryHackMe Welcome To TryHackMe.com

对每行的说明：

第 1 行： HTTP/1.1 是服务器使用的 HTTP 协议版本，后接 HTTP 状态码 “200 OK”，表示请求成功。
第 2 行： 服务器的软件类型及版本号。
第 3 行： 服务器的当前日期、时间和时区。
第 4 行： Content-Type 告知客户端即将发送的数据类型。
第 5 行： Content-Length 告知客户端响应数据长度，以确认数据无缺失。
第 6 行： HTTP 响应包含一个空行，表示响应结束。
第 7-14 行： 所请求的内容，本例中是主页。

HTTP 方法

HTTP 方法是客户端在发起 HTTP 请求时用来表示其预期操作的一种方式。HTTP 方法有很多种，但我们将介绍最常用的几种，尤其是 GET 和 POST 方法。

GET 请求
用于从网页服务器获取信息。

POST 请求
用于向网页服务器提交数据，并可能创建新的记录。

PUT 请求
用于向网页服务器提交数据以更新信息。

DELETE 请求
用于从网页服务器删除信息或记录。


HTTP 状态码

HTTP 状态码
在上一个任务中，你了解到当 HTTP 服务器响应时，第一行总是包含一个状态码，用于告知客户端其请求的结果以及如何处理。这些状态码可以分为五个不同的范围：

100-199 - 信息响应
告知客户端请求的第一部分已被接受，客户端应继续发送其余请求。这类状态码如今已不常见。

200-299 - 成功
表示客户端的请求已成功完成。

300-399 - 重定向
用于将客户端请求重定向到其他资源，可能是不同网页或不同网站。

400-499 - 客户端错误
用于告知客户端请求中存在错误。

500-599 - 服务器错误
表示服务器端出现问题，通常是服务器在处理请求时遇到较严重的问题。

常见的 HTTP 状态码
HTTP 状态码种类繁多，且应用程序可以自定义状态码。以下是一些常见的 HTTP 响应状态码：

200 - OK
请求成功完成。

201 - Created（已创建）
已创建一个资源，例如新用户或新博客文章。

301 - Moved Permanently（永久移动）
将客户端浏览器重定向到新页面，或通知搜索引擎页面已移动到其他位置。

302 - Found（临时移动）
类似于永久重定向，但这是一个临时更改，可能会在不久的将来再次更改。

400 - Bad Request（错误请求）
告知浏览器请求中存在错误或缺少内容，有时用于指示服务器资源需要某个特定参数，而客户端未发送。

401 - Not Authorized（未授权）
您无权查看此资源，直到您通过用户名和密码进行授权。

403 - Forbidden（禁止访问）
无论是否登录，您都没有权限访问该资源。

405 - Method Not Allowed（方法不允许）
资源不允许使用此方法，例如发送 GET 请求到 /create-account 路径，而该路径需要的是 POST 请求。

404 - Page Not Found（页面未找到）
请求的页面或资源不存在。

500 - Internal Service Error（服务器内部错误）
服务器遇到未知的错误，无法正常处理请求。

503 - Service Unavailable（服务不可用）
服务器无法处理请求，可能是因为过载或正在维护。

点击右侧的“查看网站”按钮，可以在浏览器中查看部分 HTTP 状态消息的示例。

请求头

请求头部信息（Headers）
请求头部信息是您在发起请求时可以发送给网页服务器的附加数据。

虽然发起 HTTP 请求时并不需要强制发送任何头部信息，但如果没有它们，您会发现很难正常查看一个网站。

常见的请求头
以下是客户端（通常是您的浏览器）发送给服务器的一些常见请求头。

Host（主机）： 一些服务器托管多个网站，通过提供 Host 头部信息，您可以告诉服务器您需要访问哪个网站，否则将收到服务器的默认网站。

User-Agent（用户代理）： 这是您的浏览器软件及其版本号，告知服务器浏览器信息可以帮助其更好地为您的浏览器格式化网站内容。此外，某些 HTML、JavaScript 和 CSS 元素仅在特定浏览器中可用。

Content-Length（内容长度）： 当您向网页服务器发送数据（例如表单数据）时，Content-Length 告诉服务器请求中包含的数据量，确保数据完整无缺。

Accept-Encoding（接受编码）： 告知服务器浏览器支持的压缩方法，以便数据可以被压缩，从而减少网络传输的数据量。

Cookie（Cookie）： 发送到服务器的数据，用于帮助记住您的信息（详细信息见 cookies 任务）。

常见的响应头
以下是服务器在响应请求时返回给客户端的一些常见响应头。

Set-Cookie（设置 Cookie）： 存储信息，每次请求时都会将其发送回服务器（详细信息见 cookies 任务）。

Cache-Control（缓存控制）： 指定在浏览器缓存中存储响应内容的时长，到期后浏览器会再次请求内容。

Content-Type（内容类型）： 告知客户端返回的数据类型，例如 HTML、CSS、JavaScript、图片、PDF、视频等。通过 Content-Type 头部信息，浏览器可以正确处理数据。

Content-Encoding（内容编码）： 指定用于压缩数据的方法，以便在网络传输时减少数据量。

Cookies（客户端存储）

可能听说过 cookies，它们是一小段存储在电脑上的数据。当从网页服务器收到一个“Set-Cookie”头部时，cookies 就会被保存下来。之后，每当再次发出请求时，cookie 数据会回传给服务器。由于 HTTP 是无状态的（不会记住的上一次请求），cookies 可以用于帮助服务器识别身份、记录网站的个性化设置，或判断是否曾访问过该网站。以下是一个包含 cookies 的 HTTP 请求示例：

Cookies 有许多用途，但最常用于网站认证。通常，cookie 的值不会是明文的密码，而是一个令牌（即一种难以被人猜到的独特代码）。

查看 Cookies
可以通过浏览器的开发者工具轻松查看浏览器发送到网站的 cookies。如果不确定如何打开开发者工具，请点击本任务顶部的“查看网站”按钮，获取相关指南。

打开开发者工具后，点击“Network”（网络）选项卡。此选项卡将显示浏览器请求的所有资源列表。可以点击任意一项，查看请求和响应的详细信息。如果浏览器发送了 cookie，可以在请求的“Cookies”选项卡中看到这些内容。
