什么是 HTTP(S)？

什么是超文本传输协议（HTTP）？

HTTP 是浏览网站时使用的通信协议，由蒂姆·伯纳斯-李及其团队在 1989 至 1991 年间研发。它定义了 Web 客户端，如浏览器与服务器之间交换网页数据的规则，用于传输 HTML、图片、视频等内容。

什么是超文本传输安全协议（HTTPS）？

HTTPS 是 HTTP 的安全版本，通过加密保护数据传输，防止他人查看发送和接收的数据。HTTPS 能确保与合法的 Web 服务器连接，而不是被伪装的服务器。

请求和响应

当访问一个网站时，浏览器向 Web 服务器发送请求，以获取 HTML、图片等资源，并接收服务器返回的响应。浏览器需要通过 URL 知道如何及从何处获取资源。

什么是统一资源定位符（URL）？

URL 是一组指令，用于指定如何在互联网上访问特定资源。每次请求的 URL 可能包含不同的组成部分。

URL 的各个部分如下所示，不一定每次请求都包含所有部分：

完整的 URL 示例：http://user:password@cyphersun.com:80/blog?id=1#tag3

http 是协议（scheme）
user 是用户信息（user）
cyphersun.com 是域名或主机（host）
80 是端口（port）
blog 是路径（path）
?id=1 是查询字符串（query string）
#tag3 是片段（fragment）

协议（Scheme）：用于指定访问资源的协议，例如 HTTP、HTTPS、文件传输协议（FTP）。
用户信息（User）：某些服务需要身份验证，可以在 URL 中包含用户名和密码。
主机（Host）：表示服务器的域名或 IP 地址，即希望访问的资源所在的位置。
端口（Port）：连接时使用的端口号，HTTP 默认端口为 80，HTTPS 为 443，也可以是 1 至 65535 之间的其他端口。
路径（Path）：表示要访问的资源的文件名或所在位置。
查询字符串（Query String）：向请求路径传递的附加信息，例如 /blog?id=1 表示请求 ID 为 1 的博客文章。
片段（Fragment）：引用页面内的特定位置，通常用于帮助用户直接跳转到较长内容页面的某一部分。

HTTP 方法

HTTP 方法是客户端在发起 HTTP 请求时，用来表示其预期操作的一种方式。HTTP 最常用的方法主要包括 GET 和 POST

GET 请求
用于从 Web 服务器获取信息，通常用于请求数据，而不对服务器上的数据进行任何修改。GET 请求一般不包含请求体，所有参数都包含在 URL 中。

POST 请求
用于向 Web 服务器提交数据，通常用于创建新资源或提交表单信息。POST 请求的参数通常放在请求体中，使其适合发送较大的数据量。

PUT 请求
用于向 Web 服务器提交数据以更新已有资源。PUT 请求通常用于完整替换目标资源，如果资源不存在，部分服务器可能会创建新资源。

DELETE 请求
用于从 Web 服务器删除指定资源或记录。DELETE 请求一般会直接从服务器上移除资源，无法撤销，需谨慎使用。

HTTP 状态码

当 HTTP 服务器响应请求时，第一行通常包含状态码，用于告知客户端请求的结果及后续处理方式。

状态码分为以下 5 个范围：

100-199 - 信息响应：表示请求的初步信息已被接受，客户端应继续发送剩余请求，现已较少使用。
200-299 - 成功：表示请求已成功完成。
300-399 - 重定向：将客户端的请求重定向至其他资源，可能是不同页面或不同网站。
400-499 - 客户端错误：表示客户端请求有误。
500-599 - 服务器错误：表示服务器在处理请求时遇到问题。

常见 HTTP 状态码

应用程序可以定义自定义 HTTP 状态码

以下是常见的 HTTP 响应：

200 OK：请求成功。
201 Created：新资源已创建，如新用户或新文章。
301 Moved Permanently：永久重定向，通常用于页面永久迁移。
302 Found：临时重定向，资源可能会再次变更位置。
400 Bad Request：请求格式有误，可能缺少必要信息。
401 Unauthorized：未授权访问，需要登录认证。
403 Forbidden：禁止访问该资源，授权也无效。
404 Not Found：资源不存在或路径错误。
405 Method Not Allowed：请求方法不被允许，例如对只支持 POST 的路径发送了 GET 请求。
500 Internal Server Error：服务器遇到意外错误，无法处理请求。
503 Service Unavailable：服务器当前无法处理请求，可能由于过载或维护中。


HTTP 头部

头部信息是在发起请求时可以发送给 Web 服务器的附加数据。虽然在 HTTP 请求中不强制包含头部信息，但缺少这些信息可能导致网页显示不正常。

常见的请求头部

请求头部是客户端浏览器发送给服务器的附加信息，帮助服务器正确理解和处理请求。

Host：指定要访问的具体网站，有些 Web 服务器托管了多个网站，Host 头部用于告知服务器具体的访问目标，否则会默认返回服务器的主站点。
User-Agent：提供浏览器软件及版本信息，便于服务器适配网页，以兼容特定浏览器的 HTML、JavaScript 和 CSS 特性。
Content-Length：当提交表单等数据时，Content-Length 头部用于告知服务器请求中的数据长度，以便确保数据完整传输。
Accept-Encoding：指定浏览器支持的压缩方式，使数据在网络上传输时更加高效。
Cookie：包含客户端的用户信息，用于帮助服务器记住用户偏好和会话数据。

常见的响应头部

响应头部是服务器在回应客户端请求时提供的额外信息，帮助客户端处理返回的数据。

Set-Cookie：用于存储信息，客户端在后续请求中会带上这些数据。
Cache-Control：指定浏览器缓存响应内容的时间，以减少频繁请求，提升性能。
Content-Type：告知客户端返回数据的类型，例如 HTML、CSS、JavaScript、图片、PDF、视频等，以便浏览器正确处理这些内容。
Content-Encoding：告知客户端数据的压缩方法，使传输更为高效。

Cookies

Cookie 是一小块存储在计算机上的数据。当 Web 服务器在响应中发送 Set-Cookie 头部时，Cookie 会被保存。之后每次请求该服务器时，浏览器会自动发送这些 Cookie 数据回给服务器。由于 HTTP 是无状态的，不会记住之前的请求，Cookie 可以帮助 Web 服务器识别用户、记住个性化设置，或确认用户是否曾访问过该网站。

Cookie 最常见的是用于网站身份验证。通常 Cookie 值不会直接包含密码，而是使用一个唯一的令牌来标识用户。

查看 Cookies

可以通过浏览器的开发者工具查看浏览器向网站发送的 Cookie。F12 打开开发者工具后，点击网络标签，这会显示浏览器请求的所有资源列表。点击任一资源可以查看请求和响应的详细信息。如果请求中包含 Cookie，可以在请求的 Cookies 标签中找到这些数据。
